# 毕业论文写作工作流

本文档面向首次接触毕业论文项目的协作者，说明如何把长期写作、审阅、回填和验收拆成可追踪的公开流程。这里不保存具体论文内容，也不记录任何私有执行历史；正式正文始终只以项目中的 LaTeX 源文件为准。

## 目标

- 让新协作者快速理解“问题来源、修改裁定、正文回填、验证交付”的职责边界。
- 让每轮修改都有记录、有状态、有验收，不依赖长对话或个人记忆。
- 让学校模板差异继续留在公共包和项目薄封装中，写作流程只处理正文维护过程。

## 五步流程

```text
scan -> adjudicate -> apply -> verify -> deliver
```

1. `scan`：只读检查目标文件，记录候选问题、位置、类型和建议动作。
2. `adjudicate`：把候选问题分为“直接处理”“需人工确认”“延后处理”，并说明原因。
3. `apply`：只在正式 LaTeX 源文件中小批量回填已裁定修改；过程记录不得成为正文来源。
4. `verify`：执行目标检索、diff 检查和论文构建命令；涉及图表编号时同步检查引用、标签和图表环境参数。
5. `deliver`：更新本轮摘要、未决项和人工审阅记录；若后续继续修改，追加新轮次，不覆盖旧记录。

## 目录职责

| 层级 | 职责 |
|---|---|
| 正文源文件 | 唯一正文真相源，所有正式修改只回填到这里 |
| 候选问题记录 | 保存扫描结果和审阅建议，不直接改正文 |
| 裁定记录 | 保存处理级别、原因和下一步动作 |
| 运行记录 | 保存本轮目标、输入范围、批次、验证结果和剩余事项 |
| 交付记录 | 保存阶段摘要、人工审阅入口和待裁定事项 |

默认建议把这些记录放在项目内的隐藏工作目录中；公开仓库只提交空模板和流程说明，不提交真实记录。

## 角色边界

- 主控角色负责最终裁定、正文回填、验证和提交。
- 审查角色只输出候选问题，不直接修改正文。
- 人工审阅意见先编号和分类，再进入正文回填。
- 长任务每完成一个批次先更新运行记录，再继续下一批。

## 记录规则

- 每轮运行记录只写本轮轨迹，不复制完整标准流程。
- 人工审阅记录只写本轮意见、处理状态和回流结果。
- 待裁定事项必须有状态：`pending`、`accepted`、`rejected`、`deferred`。
- 稳定经验可以沉淀为通用规则；具体正文、术语、图表和研究材料不得进入公开流程文件。

## 验证门禁

进入下一批修改前，至少完成：

- 目标检索：确认待处理表达、引用或标签的剩余命中。
- diff 检查：确认本批只修改预期文件，且没有格式问题。
- 构建检查：确认论文仍可生成目标产物。
- 复读检查：触及章首、结果段、讨论段、小结或摘要时，人工复读修改后的相邻段落。

涉及图表插入、删除或移动时，额外检查：

- 正文引用是否同步。
- 图表环境标签是否唯一。
- 图表文件名和说明是否与编号一致。
- 下游引用是否需要级联更新。

## UCAS 类项目适配

UCAS 类学位论文项目通常包含较多前置材料、章节源文件、图表和构建约束。使用本流程时，保持三条原则：

- 项目模板和版式问题优先回到公共包或项目入口处理。
- 正文写作问题只回填到正式 LaTeX 源文件。
- 过程记录只保留状态、决策和验证结果，不保留具体正文、真实术语、真实图表编号或人工意见原文。

## 不适用内容

本流程不公开、不迁移、不示范以下内容：

- 真实论文正文、章节改写前后对照、具体术语表、真实图表标签。
- 身份信息、评审材料、研究对象、数据来源、地点和研究细节。
- 真实日志、真实运行记录、真实人工审阅意见、真实检索结果。
- 过程性二进制文件、截图、导出文档、缓存和临时构建产物。
