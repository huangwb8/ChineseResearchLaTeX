# thesis-writing-workflow

## 适用场景

- 用户需要为学位论文建立可追踪的长期写作、审阅和交付流程。
- 用户给出一批人工审阅意见，需要先记录、分类、裁定，再回填正文。
- 用户准备对某个章节或摘要做小批量修改，并希望保留验证证据。
- 用户新增、删除或移动图表，需要检查引用、标签和编号级联。

**不适用**：生成真实论文正文；公开具体研究内容；保存真实运行日志；记录具体工具名、平台名、模型名、本机路径或私有仓库信息。

## 输入

- `<project-dir>`：毕业论文项目目录。
- `<source-files>`：本轮允许读取或修改的正式 LaTeX 源文件。
- `<review-input>`：人工审阅意见、候选问题清单或待处理任务摘要。
- `<run-log>`：本轮运行记录文件；没有时先从模板复制。
- `<verification-command>`：项目约定的论文构建或检查命令。

## 输出

- 更新后的正式 LaTeX 源文件。
- 本轮运行记录：目标、输入范围、裁定、批次、验证结果和下一步。
- 人工审阅记录：`<review-id>`、问题类型、处理状态和回流结果。
- 待裁定清单：只记录问题摘要和决策状态，不粘贴真实正文。
- 交付摘要：本轮完成项、未决项和验证状态。

## 工作流

### 1. Scan

只读检查目标范围，输出候选问题。每条候选问题至少包含：

```text
location: <file-or-section>
issue_type: <style|structure|citation|figure-table|consistency|format>
severity: <high|medium|low>
summary: <issue-summary>
suggested_action: <apply|needs-review|deferred>
```

扫描阶段不得修改正文，不输出完整段落，不记录真实写作内容。

### 2. Adjudicate

主控角色逐条裁定候选问题：

- `apply`：可直接小批量回填。
- `needs-review`：涉及科学含义、结构去留或证据边界，先进入待裁定清单。
- `deferred`：与本轮目标无关，记录原因后延后处理。

裁定结果必须写入运行记录，再进入正文修改。默认建议把运行记录、人工审阅记录和待裁定清单放在项目内的隐藏工作目录中；公开仓库只提交空模板和流程说明。

### 3. Apply

只修改正式 LaTeX 源文件。执行规则：

- 每批修改范围尽量小，避免跨多个不相关章节。
- 不把过程记录、候选清单或人工意见原文复制进正文。
- 修改涉及摘要、标题、图表编号或结论收束时，追加一致性检查。
- 审查角色只给候选问题；主控角色负责最终改文。

### 4. Verify

每批修改后执行验证：

- 目标检索：确认本批目标是否仍有残留。
- diff 检查：确认只改预期文件，且没有格式问题。
- 构建检查：确认论文仍能生成目标产物。
- 复读检查：对高压缩区域和相邻段落做人工复读。

图表变更还需检查：

- 正文引用。
- 图表标签。
- 图表环境参数。
- 下游编号级联。

### 5. Deliver

交付阶段更新三类记录：

- 运行记录：本轮批次、验证结果和下一步。
- 待裁定清单：仍需人工确认的问题。
- 交付摘要：本轮完成项、边界和验收状态。

若同一章节后续继续修改，必须追加新轮次，不覆盖旧记录。

## 公开边界

写入仓库的 workflow 或模板只能包含占位符和抽象规则。禁止写入：

- 真实论文正文、真实章节案例、具体术语表、真实图表编号或标签。
- 身份信息、评审材料、研究对象、数据来源、地点和研究细节。
- 真实日志、真实运行记录、真实人工审阅意见、真实检索结果。
- 具体工具名、平台名、模型名、本机路径或私有仓库信息。
- 过程性二进制文件、截图、导出文档、缓存和临时构建产物。

## 与其他能力的关系

本技能只负责编排：路由、记录、裁定、验证和交付门禁。若任务需要写作润色、引用核查、资料整理或规则沉淀，调用项目中已有的对应能力；本技能不重复实现这些能力。

### 路由建议

- 表达、衔接、结构或章节角色问题：交给写作修订能力处理，回填前仍由主控裁定。
- 引用、参考文献或证据链问题：交给引用核查能力处理，只把结论摘要写入记录。
- 文献背景、研究进展或指南沉淀问题：交给资料整理或规则沉淀能力处理，不在本流程保存真实检索结果。
- 图表、编号、标签和交叉引用问题：交给格式与级联检查能力处理，再由本流程安排小批量回填和验证。
- 构建、导出或版式验收问题：交给项目官方构建和检查入口处理，记录命令类别、结果和失败摘要即可。

## 快速执行清单

1. 确认 `<source-files>` 是本轮唯一允许修改的正文源。
2. 从模板建立 `<run-log>`。
3. 执行扫描，生成候选问题清单。
4. 主控裁定每条候选问题。
5. 小批量回填正式源文件。
6. 执行目标检索、diff 检查和构建检查。
7. 更新运行记录、待裁定清单和交付摘要。
