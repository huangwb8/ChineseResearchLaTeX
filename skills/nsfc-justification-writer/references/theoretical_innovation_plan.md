# 理论创新导向优化计划（执行摘要）

## 背景

立项依据是 NSFC 申请书中**最注重理论创新**的部分，评审人关注科学问题的可证伪性、理论贡献的清晰性、验证维度的完备性，而非工程落地细节。

## 优化目标

**默认支持"理论创新导向"风格**，AI 写作时自动优先考虑理论层面，而非工程落地。

## 优化方案

### 核心设计

**通过新增一个 references 文件 (`theoretical_innovation_guidelines.md`) 来约束 AI 行为**，与技能其他部分协调配合。

## 四段叙事闭环（理论创新版）

### 对比：工程导向 vs 理论创新导向

| 维度 | 工程导向 | 理论创新导向（默认） |
|------|----------|----------------------|
| **价值与必要性** | "现有系统时延高、成本大" | "现有理论框架存在因果缺失/表征不足/假设过强" |
| **现状与不足** | "准确率仅 75%，难以满足临床需求" | "依赖过强假设/缺乏统一框架/因果机制尚未阐明/最坏情形界不紧" |
| **科学问题/假说** | "将开发一个高效算法，提升性能 20%" | "通过引入 XXX 表征，可以在弱假设下证明 ZZZ 性质" |
| **项目切入点** | "将开发一套完整的软件系统" | "拟从 XXX 理论切口入手，建立 YYY，形成 ZZZ 理论框架/方法学" |
| **验证方式** | 实验结果/工程指标 | 定理证明/理论界/排除反例 |

### 关键术语映射

| 工程导向术语 | 理论创新导向术语 |
|-------------|------------------|
| 性能瓶颈/成本高 | 理论空白/认知缺失/范式不统一 |
| 准确率/时延/吞吐 | 假设过强/框架不统一/因果缺失/界不紧 |
| 开发/实现/部署 | 证明/阐明/建立/表征 |
| 系统/平台/原型 | 理论/定理/方法学/框架 |
| 实验数据/用户测试 | 理论证明/数值验证/对照实验 |

## AI 写作检查清单

AI 在生成/修改立项依据时，默认检查：

1. **价值与必要性**是否聚焦"理论空白/认知缺失"而非"性能瓶颈"？
2. **现状与不足**是否使用"假设过强/框架不统一/因果缺失/界不紧"等理论术语？
3. **核心假说**是否为"可证伪表述"而非"工程目标"？
4. **关键科学问题**是否指向"理论阐明/证明/验证"而非"开发/实现"？
5. **项目切入点**是否承诺"理论贡献/方法学"而非"系统/平台"？
6. **验证方式**是否包含"理论证明/定理/数值验证"而非仅有"实验数据"？

## 工程导向项目的处理

对于确实偏向工程应用的项目，AI 可以：

1. 仍以理论创新为主视角撰写立项依据
2. 在"现状与不足"中补充：理论缺失导致工程应用受限
3. 在"项目切入点"中说明：理论突破将为工程应用提供基础
4. 将工程细节留到"研究内容/技术路线"章节展开

## 协调机制

### 与现有技能组件的协调

| 组件 | 协调方式 | 无需修改原因 |
|------|----------|-------------|
| 内容维度覆盖检查 | 四段维度通用 | "价值/现状/科学问题/切入点"适用于理论创新导向 |
| 吹牛式表述识别 | "可验证"包含理论可验证 | 理论证明/定理/数值验证都是可验证形式 |
| 常用句式 | AI 优先使用理论创新表述 | 新增理论创新导向句式模板 |
| 信息表 | 引导用户提供理论层面信息 | 修改提问引导理论思维 |

### 写作教练示例

**修改前**：
```
- 现有方案的 2–4 条瓶颈分别是什么（尽量可量化）？
- 你准备如何验证（数据/指标/对照/消融）？
- 本项目相对现有工作的差异化切口是什么？
```

**修改后**：
```
- 现有方案在**理论层面**的 2–4 条瓶颈分别是什么（如假设过强/框架不统一/因果缺失/界不紧）？
- 你准备如何验证（理论证明/定理/数值验证/对照实验）？
- 本项目相对现有工作的**理论差异化切口**是什么（如新表征/新方法学/统一框架）？
```

### 评审建议示例

**修改前**：
```
- 现有方案的瓶颈是否是 2–4 条"可量化/可验证"的不足，而不是口号？
```

**修改后**：
```
- 现有方案在**理论层面**的瓶颈是否是 2–4 条"可验证"的不足（如假设过强/框架不统一/因果缺失/界不紧），而不是口号？
```
