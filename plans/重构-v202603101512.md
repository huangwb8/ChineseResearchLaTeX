# NSFC LaTeX 模板重构计划

**版本**: v202603101512-optimized  
**创建时间**: 2026-03-10 15:12  
**更新时间**: 2026-03-10 16:28  
**目标**: 将 `projects` 目录下的共性配置抽取为 `nsfc-common` LaTeX 包，采用**分层设计**保证灵活性，并提供 Python 脚本管理包的安装和更新。

---

## 一、项目背景

### 当前问题

1. **代码重复**：三个项目（General/Local/Young）的 `@config.tex` 大量内容相同。
2. **维护成本高**：修改配置时需要同步更新多个文件。
3. **一致性风险**：人工同步容易遗漏，导致模板行为不一致。
4. **用户诉求分层**：既有想“一行可用”的新手，也有想“深度接管”的高级用户。

### 重构目标

1. **DRY 原则**：消除重复配置，建立单一真相来源。
2. **易维护**：共性配置集中维护，项目差异可控覆盖。
3. **保持灵活性**：用户可在“开箱即用”和“深度定制”之间自由选择。
4. **用户友好**：提供类似 `pip install` 的安装体验。
5. **向后兼容**：不破坏现有用户的使用习惯，支持渐进式迁移。
6. **可验证**：重构后可通过编译与 PDF 对比验证版式一致性。

## 二、技术方案

### 2.1 设计哲学：分层架构

采用**三层配置架构**，在提升可维护性的同时保持用户的自定义灵活性：

```text
┌─────────────────────────────────────────────────────┐
│ 第 3 层：项目自定义层 (@config.tex)                 │
│ - 优先级：最高                                      │
│ - 用途：承接项目差异与少量必要覆盖                  │
└─────────────────────────────────────────────────────┘
                        ↓ 覆盖
┌─────────────────────────────────────────────────────┐
│ 第 2 层：项目类型层 (nsfc-common 包选项)            │
│ - 优先级：中等                                      │
│ - 用途：按 general/local/young 应用差异化配置       │
└─────────────────────────────────────────────────────┘
                        ↓ 继承
┌─────────────────────────────────────────────────────┐
│ 第 1 层：基础配置层 (nsfc-common 核心)              │
│ - 优先级：最低                                      │
│ - 用途：提供共性配置（宏包、字体、行距等）          │
└─────────────────────────────────────────────────────┘
```

### 2.2 使用原则

本次重构改为**单一路径**，不再提供“完全包 / 最小包 / 混合 / 独立”四种模式：

- 用户继续沿用现有入口：`main.tex -> extraTex/@config.tex`
- `@config.tex` 从“完整配置大文件”收敛为“项目薄封装”
- 共性配置统一下沉到 `nsfc-common.sty`
- 项目差异仅保留在各自 `@config.tex` 中

这样可以避免模式分叉、降低测试矩阵，并保持现有用户的使用习惯不变。

### 2.3 架构设计

```text
ChineseResearchLaTeX/
├── common/
│   ├── nsfc-common.sty          # LaTeX 包源文件（核心）
│   ├── package.json             # 包元信息（供安装器/发布流程使用）
│   ├── install.py               # 安装/更新脚本
│   ├── README.md                # 包使用说明
│   └── examples/
│       └── basic-usage.tex
├── projects/
│   ├── NSFC_General/
│   │   ├── main.tex
│   │   └── extraTex/
│   │       └── @config.tex
│   ├── NSFC_Local/
│   └── NSFC_Young/
└── scripts/
    └── validate_package.py
```

### 2.4 包设计

#### 核心原则

- `nsfc-common.sty` 负责**共性能力 + 项目类型差异化分发**。
- `@config.tex` 仍然保留，但职责收敛为项目层薄封装。
- 不再额外引入 `@config-override.tex`、模式切换开关或多套入口。
- 优先保证默认路径简单稳定，而不是追求“所有层都可自由拼装”。

#### 选项系统

为了保留你原来设计的 `type=general` 用法，包内部建议显式采用 `kvoptions`（或等效 key-value 方案）实现，而不是只用基础 `\DeclareOption`。

建议支持的选项：

- `type=general|local|young`

这样既保留原设计中的项目类型分发能力，又避免把“灵活性”做成额外复杂度。

#### 钩子与覆盖策略

保留用户覆盖钩子，但控制数量，优先保留以下几个：

- `\NSFCBeforePackages`
- `\NSFCAfterPackages`
- `\NSFCBeforeFonts`
- `\NSFCAfterFonts`
- `\NSFCUserOverride`

优化约束：

1. 钩子数量控制在最小必要集，避免“处处可插”的维护负担。
2. 对普通用户，不额外暴露“模式选择”；默认只维护 `@config.tex` 这一个项目入口。
3. 文档里只说明必要钩子，不把钩子包装成新的使用流派。

#### `nsfc-common.sty` 结构建议

```latex
\NeedsTeXFormat{LaTeX2e}
\ProvidesPackage{nsfc-common}[2026/03/10 v1.0.0 NSFC Common Package]

\RequirePackage{kvoptions}
\SetupKeyvalOptions{family=NSFC,prefix=NSFC@}

% 项目类型
\DeclareStringOption[general]{type}

\ProcessKeyvalOptions*

% 用户覆盖钩子
\providecommand{\NSFCBeforePackages}{}
\providecommand{\NSFCAfterPackages}{}
\providecommand{\NSFCBeforeFonts}{}
\providecommand{\NSFCAfterFonts}{}
\providecommand{\NSFCUserOverride}{}

% 基础配置
\NSFCBeforePackages
% ... 加载共性宏包
\NSFCAfterPackages

% 字体配置
\NSFCBeforeFonts
% ... 字体逻辑
\NSFCAfterFonts

% 项目差异化配置
\ifthenelse{\equal{\NSFC@type}{general}}{...}{}
\ifthenelse{\equal{\NSFC@type}{local}}{...}{}
\ifthenelse{\equal{\NSFC@type}{young}}{...}{}

% 用户最终覆盖
\NSFCUserOverride
```

说明：

- 这里不强求逐行照抄实现，重点是**让示例语法与实现机制一致**。
- 版式敏感部分不要为了“抽得更干净”而硬统一，必要时仍留在项目层 `@config.tex` 中。

### 2.5 使用示例

#### 推荐接入方式

```latex
\documentclass[12pt,UTF8,AutoFakeBold=2.5,a4paper,fontset=none]{ctexart}
\input{extraTex/@config.tex}

\begin{document}
% 内容
\end{document}
```

其中 `extraTex/@config.tex` 推荐收敛为：

```latex
\usepackage[type=general]{nsfc-common}
% 仅保留当前项目的少量差异化配置
```

### 2.7 Python 包管理器

#### 核心功能

1. **安装**：`python common/install.py install`
   - 检测 `TEXMFHOME`
   - 复制 `.sty` 文件到 `$TEXMFHOME/tex/latex/nsfc-common/`
   - 刷新文件数据库（如需要）
   - 保存版本信息

2. **升级**：`python common/install.py upgrade`
   - 检查当前版本与已安装版本
   - 备份旧版本
   - 安装新版本

3. **卸载**：`python common/install.py uninstall`
   - 删除安装的包文件
   - 清理版本信息

4. **状态查询**：`python common/install.py status`
   - 显示当前版本、已安装版本、安装路径

#### 技术细节

- **跨平台支持**：自动检测 Windows/macOS/Linux
- **版本管理**：在安装目录保存 `.version`
- **备份机制**：升级前备份到 `.backup`
- **错误处理**：提供清晰错误提示与回滚建议
- **兼容策略**：即便安装器不可用，仓库内模板仍保留现有入口结构，可通过仓库内分发方式直接编译

## 四、风险评估与应对

### 4.1 技术风险

| 风险 | 影响 | 概率 | 应对措施 |
|------|------|------|----------|
| `type=general` 示例与实现不一致 | 高 | 中 | 显式引入 key-value 选项系统，统一示例与实现 |
| 钩子机制过多导致维护复杂 | 中 | 中 | 保留最小必要钩子集，不再额外设计 override 分支 |
| 公共包抽取过度导致版式回归 | 高 | 中 | 先抽稳定共性，版式敏感项继续留在项目层 |
| 仓库内直编与安装器分发路径不一致 | 高 | 中 | 在首轮实现中先固定仓库内分发策略，再补安装器适配 |
| 跨平台字体差异影响 PDF 对比 | 中 | 中 | 将像素级比对限定在同环境，跨平台采用结构性校验 |

### 4.2 用户体验风险

| 风险 | 影响 | 概率 | 应对措施 |
|------|------|------|----------|
| 用户不愿安装 Python 脚本 | 中 | 中 | 保留手动安装说明与仓库内直编路径 |
| 用户担心失去灵活性 | 中 | 中 | 保留项目层 `@config.tex`，只抽取稳定共性 |
| 用户默认路径被打断 | 高 | 低 | 保持 `main.tex -> @config.tex` 入口不变 |

### 4.3 维护风险

| 风险 | 影响 | 概率 | 应对措施 |
|------|------|------|----------|
| 安装器与包版本漂移 | 中 | 中 | 以 `common/package.json` 为安装器版本源，保持文档同步 |
| 过度抽象导致排错困难 | 高 | 中 | 对版式敏感项允许保留在项目覆盖层 |
| 文档与实现脱节 | 中 | 中 | 示例文件参与实际编译测试，避免只写不测 |
