# NSFC LaTeX 模板重构计划

**版本**: v202603101512-optimized  
**创建时间**: 2026-03-10 15:12  
**更新时间**: 2026-03-15 （强化 Python 安装器唯一入口与 Git 版本安装设计）  
**目标**: 将 `projects` 目录下的共性配置抽取为 `bensz-nsfc-common` LaTeX 包，采用**分层设计**保证灵活性，并以 Python 安装器作为唯一官方入口，支持基于 Git 引用安装、锁定、升级与回退任意模板版本。

---

## 一、项目背景

### 当前问题

1. **代码重复**：三个项目（General/Local/Young）的 `@config.tex` 大量内容相同。
2. **维护成本高**：修改配置时需要同步更新多个文件。
3. **一致性风险**：人工同步容易遗漏，导致模板行为不一致。
4. **用户诉求分层**：既有想“一行可用”的新手，也有想“深度接管”的高级用户。
5. **版本控制薄弱**：用户可能长期停留在某个历史模板版本，但当前缺少“按 Git 历史精确安装 / 锁定 / 回退 / 前进”的正式机制。

### 重构目标

1. **DRY 原则**：消除重复配置，建立单一真相来源。
2. **易维护**：共性配置集中维护，项目差异可控覆盖。
3. **保持灵活性**：用户可在“开箱即用”和“深度定制”之间自由选择。
4. **用户友好**：提供类似 `pip install` 的安装体验。
5. **向后兼容**：不破坏现有用户的使用习惯，支持渐进式迁移。
6. **可验证**：重构后可通过编译与 PDF 对比验证版式一致性。
7. **版本可控**：用户可通过同一个 Python 脚本固定到任意历史版本，也可随时前进到任意更新版本。

---

## 二、技术方案

### 2.1 设计哲学：分层架构

采用**三层配置架构**，在提升可维护性的同时保持用户的自定义灵活性：

```text
┌─────────────────────────────────────────────────────┐
│ 第 3 层：项目自定义层 (@config.tex)                 │
│ - 优先级：最高                                      │
│ - 用途：承接项目差异与少量必要覆盖                  │
└─────────────────────────────────────────────────────┘
                        ↓ 覆盖
┌─────────────────────────────────────────────────────┐
│ 第 2 层：项目类型层 (bensz-nsfc-common 包选项)      │
│ - 优先级：中等                                      │
│ - 用途：按 general/local/young 应用差异化配置       │
└─────────────────────────────────────────────────────┘
                        ↓ 继承
┌─────────────────────────────────────────────────────┐
│ 第 1 层：基础配置层 (bensz-nsfc-common 核心)        │
│ - 优先级：最低                                      │
│ - 用途：提供共性配置（宏包、字体、行距等）          │
└─────────────────────────────────────────────────────┘
```

借鉴已有成熟 texmf 包的经验，本次重构进一步强调以下几点：

- **用户入口保持极简**：用户仍然只接触 `main.tex -> @config.tex`
- **内部结构模块化**：按 layout / typography / headings / bibliography 等关注点拆包
- **参数与实现分离**：项目类型差异放入 profile 文件，模块只消费参数
- **版本控制前置**：以 Git 引用作为模板版本主键，由安装器负责解析、缓存、激活与锁定
- **源码/安装分离**：仓库内用 `packages/` 管理源码；安装器再把源码映射到标准 `texmf` 目录
- **分发方式标准化**：安装器统一消费 GitHub 仓库快照或 Release 附件，TDS zip 只作为辅助分发产物

### 2.2 使用原则

本次重构改为**单一路径**，不再提供“完全包 / 最小包 / 混合 / 独立”四种模式：

- 用户继续沿用现有入口：`main.tex -> extraTex/@config.tex`
- `@config.tex` 从“完整配置大文件”收敛为“项目薄封装”
- 共性配置统一下沉到 `bensz-nsfc-common.sty`
- 项目差异仅保留在各自 `@config.tex` 中
- 核心包的安装、切换、升级、回退统一通过 `python packages/bensz-nsfc/scripts/install.py` 完成

这样可以避免模式分叉、降低测试矩阵，并保持现有用户的使用习惯不变。

### 2.4 包设计

#### 核心原则

- `bensz-nsfc-common.sty` 只作为**薄入口**，避免把所有逻辑堆进一个文件。
- `bensz-nsfc-core.sty` 负责**选项解析、profile 加载、模块调度**。
- 各功能模块只负责一个关注点，避免“大一统配置文件”继续膨胀。
- 项目类型差异通过 profile 文件集中定义，形成清晰的单一真相来源。
- `@config.tex` 仍然保留，但职责收敛为项目层薄封装。
- 不再额外引入 `@config-override.tex`、模式切换开关或多套入口。
- 优先保证默认路径简单稳定，而不是追求“所有层都可自由拼装”。
- `ChineseResearchLaTeX` 保留为**项目 / 仓库名称**，不再作为任何具体源码包目录或 texmf 安装目录名。
- 具体被 `\usepackage{...}` / `\documentclass{...}` 调用的文件，建议采用带身份标识的清晰命名，如 `bensz-nsfc-*`、`bensz-paper-*`、`bensz-thesis-*`，避免未来多包时冲突。

#### 选项系统

为了保留你原来设计的 `type=general` 用法，包内部建议显式采用 `kvoptions`（或等效 key-value 方案）实现，而不是只用基础 `\DeclareOption`。

建议支持的选项：

- `type=general|local|young`

这样既保留原设计中的项目类型分发能力，又避免把“灵活性”做成额外复杂度。

#### 内部结构约束

建议将公共包内部稳定拆成以下几层：

- **入口层**：`bensz-nsfc-common.sty`
- **核心层**：`bensz-nsfc-core.sty`
- **参数层**：`profiles/bensz-nsfc-profile-*.def`
- **功能层**：`bensz-nsfc-layout.sty`、`bensz-nsfc-typography.sty`、`bensz-nsfc-headings.sty`、`bensz-nsfc-bibliography.sty`

每层职责如下：

- 入口层只暴露包名，不承载具体版式细节
- 核心层只做“读选项、载入 profile、按顺序加载模块”
- 参数层只定义数字、开关和项目类型差异，不直接写实现逻辑
- 功能层只消费参数，不重新发明项目类型判断

#### 钩子与覆盖策略

保留用户覆盖钩子，但控制数量，优先保留以下几个：

- `\NSFCBeforePackages`
- `\NSFCAfterPackages`
- `\NSFCBeforeFonts`
- `\NSFCAfterFonts`
- `\NSFCUserOverride`

优化约束：

1. 钩子数量控制在最小必要集，避免“处处可插”的维护负担。
2. 对普通用户，不额外暴露“模式选择”；默认只维护 `@config.tex` 这一个项目入口。
3. 文档里只说明必要钩子，不把钩子包装成新的使用流派。

#### `bensz-nsfc-common.sty` 结构建议

```latex
\NeedsTeXFormat{LaTeX2e}
\ProvidesPackage{bensz-nsfc-common}[2026/03/15 p_v20260315 ChineseResearchLaTeX NSFC Common Package]

\RequirePackage{bensz-nsfc-core}
```

说明：

- `bensz-nsfc-common.sty` 应尽量薄，避免再次成为“新的大杂烩”。
- 真正的公共逻辑建议下沉到 `bensz-nsfc-core.sty` 与各子模块中。
- 版式敏感部分不要为了“抽得更干净”而硬统一，必要时仍留在项目层 `@config.tex` 中。
- 方括号中的 `2026/03/15` 是 LaTeX 生态约定的发布日期字段；真正的包版本号是后面的 `p_v20260315`。

#### `bensz-nsfc-core.sty` 结构建议

```latex
\NeedsTeXFormat{LaTeX2e}
\ProvidesPackage{bensz-nsfc-core}[2026/03/15 p_v20260315 ChineseResearchLaTeX NSFC Core]

\RequirePackage{kvoptions}
\RequirePackage{etoolbox}
\SetupKeyvalOptions{family=NSFC,prefix=NSFC@}

\DeclareStringOption[general]{type}
\ProcessKeyvalOptions*

% 用户覆盖钩子
\providecommand{\NSFCBeforePackages}{}
\providecommand{\NSFCAfterPackages}{}
\providecommand{\NSFCBeforeFonts}{}
\providecommand{\NSFCAfterFonts}{}
\providecommand{\NSFCUserOverride}{}

% 载入项目 profile
\InputIfFileExists{profiles/bensz-nsfc-profile-\NSFC@type.def}{}{%
  \PackageError{bensz-nsfc-common}{Profile '\NSFC@type' not found}{}
}

% 按顺序载入模块
\RequirePackage{bensz-nsfc-layout}
\RequirePackage{bensz-nsfc-headings}
\RequirePackage{bensz-nsfc-typography}
\RequirePackage{bensz-nsfc-bibliography}
```

说明：

- 这里借鉴成熟包的做法，把“项目类型差异”与“实现逻辑”解耦。
- 如果后续需要新增 `重点项目` 等模板类型，只需新增 profile，而不是在一个大文件里继续堆 `ifthenelse`。
- 同理，这里的 `2026/03/15` 只表示发布日期，不取代 `p_v20260315` 这一包版本号。

#### Profile 设计建议

三个 profile 文件应集中保存“项目级差异参数”，例如：

- 页边距
- 标题前后距
- 参考文献条目宽度
- 是否启用 `\raggedbottom`
- 其它确实存在项目差异的版式数值

原则：

- 共性数值放核心默认值
- 项目差异数值放 profile
- 项目私有、且强依赖正文结构的内容留在 `@config.tex`

这样可以把现在三个 `@config.tex` 之间的差异，逐步重组为“参数差异”，而不是“整段代码差异”。

### 2.5 使用示例

#### 推荐接入方式

```latex

\input{extraTex/@config.tex}

```

其中 `extraTex/@config.tex` 推荐收敛为：

```latex

```

不建议在首版引入新的用户命令（如 `\NSFCSetup{...}`）作为主要入口，避免增加学习成本。

### 2.6 包元信息

- 主要用于**安装器、版本检查、发布辅助、兼容性声明**；
- 不是为了把本项目伪装成 Node 包；
- 如果后续发现 `config.yaml` 更适合，也可在实施前再统一调整，但当前计划先保持你的既有设计不变。

#### 内部版本号规则

本项目内部版本号采用**双层日期制**：

| 层次 | 格式 | 示例 | 说明 |
|------|------|------|------|
| **包版本** | `p_vYYYYMMDD` | `p_v20260315` | 标识某个 LaTeX 包本身的发布版本 |
| **模板版本** | `t_vYYYYMMDD` | `t_v20260315` | 标识包内某个具体模板的版本 |

约束：

- 模板版本**总是依赖于包版本**，不能脱离所属包单独解释。
- 同一个模板的 `t_v*` 变化，必须对应一个明确的所属 `package_name + package_version`。
- Git `tag / branch / commit` 仍然是仓库级可安装来源；`p_v*` 与 `t_v*` 是仓库内容内部的结构化版本标识。
- `\ProvidesPackage` / `\ProvidesClass` 中前置的 `YYYY/MM/DD` 仅作为 LaTeX 标准发布日期字段，不属于本项目的包版本号或模板版本号体系。

### 2.7 Python 安装器（唯一官方入口）

#### 核心定位

本次重构后，**用户安装、切换、升级、回退 `bensz-nsfc-common` 的唯一官方方式**是：

```bash
python packages/bensz-nsfc/scripts/install.py ...
```

约束如下：

- 普通用户**不再**以“手工复制 `.sty` 到 texmf”作为标准流程。
- 仓库内直编仍可保留给开发者做回归测试，但它不等同于“已正式安装模板包”。
- 所有版本控制能力都收敛到同一个安装器，避免出现“文档说能回退，实际只能覆盖安装”的断层。
- 仅靠目录名 `bensz-nsfc/` 不能避免 TeX 文件名冲突；真正参与 `\usepackage` / `\RequirePackage` 解析的是文件基名，因此对外分发文件统一加 `bensz-` 前缀。

#### 安装器职责

安装器需要承担四类职责：

1. **版本解析**
   - 接受用户指定的 Git 引用：`tag`、`branch`、`commit SHA`
   - 将可变引用（如 `main`）解析为不可变提交 SHA，供缓存与锁定使用

2. **源码获取**
   - 默认从 GitHub 仓库按 ref 下载归档快照
   - 可选消费 Release 附件（如 TDS zip）加速稳定版本安装
   - 开发者模式下支持 `--source local --path packages/bensz-nsfc` 安装当前本地工作树快照

3. **本地激活**
   - 检测 `TEXMFHOME`
   - 将 `packages/bensz-nsfc/` 中的包文件复制到 `$TEXMFHOME/tex/latex/bensz-nsfc/`
   - 刷新文件数据库（如 `mktexlsr`）
   - 记录当前激活版本与上一个版本

4. **项目对齐**
   - 读取当前项目的 `.nsfc-version`
   - 检查全局激活版本是否与项目锁定版本一致
   - 一键同步到项目要求的版本

#### 命令设计

优先保留下面这组稳定命令：

| 命令 | 说明 |
|------|------|
| `install.py install --ref v3.5.1` | 下载并缓存指定 Git 引用；默认同时激活 |
| `install.py install --ref main` | 安装某个移动引用的当前快照，并记录其解析后的 commit |
| `install.py use --ref v3.5.1` | 使用本地已缓存版本；若未缓存则先下载 |
| `install.py pin --ref v3.5.1` | 在当前项目写入锁文件 |
| `install.py sync` | 读取当前项目锁文件并切换到对应版本 |
| `install.py check` | 检查“当前激活版本”与“项目锁定版本”是否一致 |
| `install.py list` | 列出本地缓存版本 |
| `install.py list --remote` | 列出远端可见的 tag / release |
| `install.py rollback` | 回退到上一个激活版本 |
| `install.py prune` | 清理本地未使用缓存 |
| `install.py status` | 显示激活版本、来源 ref、解析 commit、安装路径 |
| `install.py uninstall` | 卸载当前激活版本并清理状态 |

说明：

- 首版不追求“大而全”命令集，重点把 `install / use / pin / sync / check / rollback` 做扎实。
- `upgrade` 不再作为核心抽象；对用户而言，**“升级到哪个版本”本质上就是 `install --ref <new-ref>`**。

#### 技术细节

- **跨平台支持**：自动检测 Windows/macOS/Linux
- **Git 依赖最小化**：优先使用 GitHub 归档接口，尽量不要求用户本机安装 Git
- **缓存优先**：相同 commit 不重复下载
- **回滚友好**：激活新版本前保留上一个激活版本指针
- **错误处理**：提供清晰错误提示与回退建议
- **开发者友好**：允许从 `packages/bensz-nsfc/` 这样的本地源码目录安装当前工作树快照，但仍复用同一安装器
- **分发标准化**：Release 可继续附带 TDS zip，但主要服务安装器与离线分发，而不是替代安装器
- **安装目录按包拆分**：当前 NSFC 包激活版本统一安装到 `$TEXMFHOME/tex/latex/bensz-nsfc/`；未来论文模板与毕业论文模板分别使用 `bensz-paper/`、`bensz-thesis/`

### 2.8 模板版本控制系统（以 Git 引用为核心）

#### 设计动机

用户对模板版本控制的真实需求不是“只装最新版”，而是：

- 某个项目可能长期停留在过去某个模板版本
- 另一个项目可能需要持续跟进最新模板
- 任意时刻都可能要回退到某个历史节点重新编译

因此，本次重构的版本控制系统必须把“**任意 Git 版本可安装**”作为第一等能力。

#### 版本语义

本计划明确区分三层版本概念：

| 层次 | 作用 | 示例 |
|------|------|------|
| **仓库版本（用户主语义）** | 用户真正指定的可安装来源 | `v3.5.1` / `main` / `9f31a7c` |
| **解析版本（可复现语义）** | 安装器解析后的不可变 commit | `9f31a7c4...` |
| **包版本（内部语义）** | 某个 LaTeX 包本身版本 | `p_v20260315` |
| **模板版本（内部语义）** | 某个具体模板版本 | `t_v20260315` |

结论：

- **用户面向 Git 引用操作**，因为这和 GitHub 托管事实一致。
- **安装器面向 commit 缓存**，因为 commit 才是可复现单元。
- **Git ref 负责“从哪里安装”**，`p_v*` 负责“包本身是什么版本”，`t_v*` 负责“包内模板是什么版本”。
- **模板版本从属于包版本**，不能只记录 `t_v*` 而不记录所属 `p_v*`。

#### 版本来源

版本来源按优先级定义为：

1. **Git tag / commit / branch**：唯一权威版本标识
2. **GitHub Releases**：稳定版本的推荐入口和说明页
3. **Release 附件 TDS zip**：稳定版本的加速分发格式

这意味着：

- 任何发布过 tag 的历史模板都应可安装
- 即便没有单独制作 Release，只要 Git ref 存在，也应能安装
- 用户要“回到某个老版本”，本质上就是 `install --ref <old-tag-or-sha>`

#### 本地存储结构

```text
~/.bensz-nsfc/
├── cache/
│   ├── commits/
│   │   ├── 9f31a7c4.../
│   │   │   ├── common/
│   │   │   └── metadata.json
│   │   └── c2b8d1e9.../
│   └── refs/
│       ├── v3.5.1.json
│       └── main.json
├── state.json
└── registry.json
```

约定：

- `cache/commits/` 以**解析后的 commit** 为唯一缓存键，避免同一版本重复下载
- `cache/refs/` 记录“用户请求的 ref -> 解析 commit”的最近一次映射
- `state.json` 记录当前激活版本、上一个版本、激活时间、安装路径
- `registry.json` 缓存最近一次远端查询结果，供离线展示
- 当前 NSFC 安装器先使用 `~/.bensz-nsfc/`；未来 `paper`、`thesis` 包各自维护独立缓存目录，避免不同包之间的状态互相污染

TEXMFHOME 中始终只保留**一个激活版本**的副本，由安装器负责切换。

#### 源码目录与安装目录

仓库源码目录与最终 texmf 安装目录应明确区分：

```text
repo source:
packages/bensz-nsfc/
packages/bensz-paper/
packages/bensz-thesis/
```

```text
installed target:
$TEXMFHOME/tex/latex/bensz-nsfc/
$TEXMFHOME/tex/latex/bensz-paper/
$TEXMFHOME/tex/latex/bensz-thesis/
```

其中当前 NSFC 包从源码到安装目标的映射为：

```text
source:
packages/bensz-nsfc/
├── bensz-nsfc-common.sty
├── bensz-nsfc-core.sty
├── bensz-nsfc-layout.sty
├── bensz-nsfc-typography.sty
├── bensz-nsfc-headings.sty
├── bensz-nsfc-bibliography.sty
└── profiles/
    ├── bensz-nsfc-profile-general.def
    ├── bensz-nsfc-profile-local.def
    └── bensz-nsfc-profile-young.def

installed:
$TEXMFHOME/tex/latex/bensz-nsfc/
├── bensz-nsfc-common.sty
├── bensz-nsfc-core.sty
├── bensz-nsfc-layout.sty
├── bensz-nsfc-typography.sty
├── bensz-nsfc-headings.sty
├── bensz-nsfc-bibliography.sty
└── profiles/
    ├── bensz-nsfc-profile-general.def
    ├── bensz-nsfc-profile-local.def

└── ... future paper packages ...

$TEXMFHOME/tex/latex/bensz-thesis/
└── ... future thesis packages ...
```

约束：

- `ChineseResearchLaTeX` 只作为项目名，不作为任何具体 texmf 文件夹名
- 仓库内优先使用 `packages/` 作为人类友好的源码目录
- `bensz-nsfc` 专门负责国自然模板
- `bensz-paper` 专门负责论文相关模板
- `bensz-thesis` 专门负责毕业论文模板
- 安装器负责把 `packages/` 下的源码映射到标准 `texmf/tex/latex/...` 安装目录
- 具体 `.sty/.cls` 文件继续采用带身份标识的命名，如 `bensz-nsfc-*`、`bensz-paper-*`、`bensz-thesis-*`

#### 项目级锁文件

每个项目目录下保留 `.nsfc-version`，但内容从“纯字符串”升级为**最小结构化锁文件**，建议使用 JSON：

```json
{
  "ref": "v3.5.1",
  "commit": "9f31a7c4f0c8...",
  "package_name": "bensz-nsfc-common",
  "package_version": "p_v20260315",
  "template_id": "general",
  "template_version": "t_v20260315"
}
```

这样设计的原因是：

- 如果用户锁的是 `main`，仅保存 `main` 不能保证可复现
- 保存 `commit` 后，未来即使 `main` 已前进，旧项目仍能精确恢复
- `package_name` + `package_version` 只是辅助排错信息，真正的锁定主键仍是 `commit`
- `template_id` + `template_version` 用于表达“当前项目实际依赖的是包中的哪个模板版本”
- 模板版本必须和所属包版本一起出现，避免出现“只知道模板版本，不知道依赖哪个包版本”的歧义

锁文件应加入 Git 版本控制，确保协作环境一致。

#### 版本命令与行为约束

##### 安装

```bash
python packages/bensz-nsfc/scripts/install.py install --ref v3.5.1
python packages/bensz-nsfc/scripts/install.py install --ref 9f31a7c
python packages/bensz-nsfc/scripts/install.py install --ref main
```

行为要求：

- 总是先解析 ref，再缓存为 commit
- 默认激活刚安装的版本
- 若目标 commit 已缓存，则直接复用

##### 锁定与同步

```bash
python packages/bensz-nsfc/scripts/install.py pin --ref v3.5.1
python packages/bensz-nsfc/scripts/install.py sync
python packages/bensz-nsfc/scripts/install.py check
```

行为要求：

- `pin` 写入 `.nsfc-version`
- `sync` 读取锁文件并把本机环境切到目标 commit
- `check` 必须输出“匹配 / 不匹配 / 未锁定”三态结果

##### 回退与前进

```bash
python packages/bensz-nsfc/scripts/install.py rollback
python packages/bensz-nsfc/scripts/install.py install --ref v3.6.0
python packages/bensz-nsfc/scripts/install.py install --ref main
```

行为要求：

- `rollback` 基于 `state.json.previous_commit`
- 所谓“升级”与“降级”都统一成“切到另一个 ref”
- 对用户不再区分“历史版本”和“未来版本”，两者都只是 Git ref

#### 典型用户工作流

**场景 1：开始一个新项目，并锁定当前稳定版本**

```bash
python packages/bensz-nsfc/scripts/install.py install --ref v3.5.1
cd projects/NSFC_General
python ../../packages/bensz-nsfc/scripts/install.py pin --ref v3.5.1
```

**场景 2：旧项目停留在老版本，新项目跟进新版**

```bash
python packages/bensz-nsfc/scripts/install.py install --ref v3.6.0
python packages/bensz-nsfc/scripts/install.py use --ref v3.6.0

cd projects/NSFC_General
python ../../packages/bensz-nsfc/scripts/install.py check
python ../../packages/bensz-nsfc/scripts/install.py sync
```

**场景 3：跟踪 `main` 一段时间，然后冻结到当前提交**

```bash
python packages/bensz-nsfc/scripts/install.py install --ref main
python packages/bensz-nsfc/scripts/install.py status
# status 显示 main -> 解析 commit = 9f31a7c4...
python packages/bensz-nsfc/scripts/install.py pin --ref main
```

这里 `pin` 写入的应是 `ref=main` + `commit=9f31a7c4...`，保证既保留“来源语义”，又保留“可复现语义”。

**场景 4：离线编译历史项目**

```bash
cd projects/NSFC_Local
python ../../packages/bensz-nsfc/scripts/install.py sync
```

若锁定 commit 已在本地缓存，则无需联网。

#### LaTeX 包内版本声明

每个版本的 `bensz-nsfc-common.sty` 仍保留包内版本号：

```latex
\ProvidesPackage{bensz-nsfc-common}[2026/03/15 p_v20260315 ChineseResearchLaTeX NSFC Common Package]
```

安装器通过解析该声明做二次校验，但**不把它当作用户主版本语义**；模板版本则由包内模板元信息或锁文件中的 `template_version` 表达。

## 四、风险评估与应对

### 4.1 技术风险

| 风险 | 影响 | 概率 | 应对措施 |
|------|------|------|----------|
| `type=general` 示例与实现不一致 | 高 | 中 | 显式引入 key-value 选项系统，统一示例与实现 |
| 钩子机制过多导致维护复杂 | 中 | 中 | 保留最小必要钩子集，不再额外设计 override 分支 |
| 公共包抽取过度导致版式回归 | 高 | 中 | 先抽稳定共性，版式敏感项继续留在项目层 |
| 模块加载顺序不当引发副作用 | 高 | 中 | 在核心层固定模块加载顺序，并用编译测试覆盖 |
| profile 参数与模块实现漂移 | 中 | 中 | 以 profile 文件为参数单一事实来源，文档同步引用 |
| 仓库内直编与安装器分发路径不一致 | 高 | 中 | 在首轮实现中先固定仓库内分发策略，再补安装器适配 |
| 用户锁定的是旧项目，但只切了包版本没切仓库语义 | 高 | 中 | 锁文件以 Git ref + commit 为主，不只记录 `p_v*` 包版本 |
| 跨平台字体差异影响 PDF 对比 | 中 | 中 | 将像素级比对限定在同环境，跨平台采用结构性校验 |
| 版本切换后 TEXMFHOME 缓存未刷新导致旧版本仍生效 | 高 | 中 | 切换时强制执行 `mktexlsr`，并在编译前提示用户验证激活版本 |
| GitHub API 限流或不可达导致版本拉取失败 | 中 | 中 | 离线降级策略：优先使用本地 `registry.json` 缓存，并给出友好提示 |
| 分支名（如 `main`）是可变引用，导致复现失败 | 高 | 中 | 锁文件必须保存解析后的 commit；`check` 同时展示 ref 与 commit |
| `.nsfc-version` 锁文件与实际 TEXMFHOME 版本不一致 | 高 | 中 | `install.py check` 在编译前自动检测并警告，避免静默编译错误版本 |
| 多项目并发使用不同版本（全局只有一个激活版本）| 中 | 高 | 文档明确说明全局单激活限制；进阶方案提供项目本地 texmf 目录覆盖路径 |

### 4.2 用户体验风险

| 风险 | 影响 | 概率 | 应对措施 |
|------|------|------|----------|
| 用户不愿安装 Python 脚本 | 中 | 中 | 明确这是唯一官方入口；同时把脚本尽量做成免 Git、低依赖、跨平台 |
| 用户担心失去灵活性 | 中 | 中 | 保留项目层 `@config.tex`，只抽取稳定共性 |
| 用户默认路径被打断 | 高 | 低 | 保持 `main.tex -> @config.tex` 入口不变 |
| 版本控制概念对新手过于复杂 | 中 | 中 | 默认推荐“稳定 tag + pin + sync”三步法，其它 Git ref 作为进阶功能文档化 |

### 4.3 维护风险

| 风险 | 影响 | 概率 | 应对措施 |
|------|------|------|----------|
| 安装器与包版本漂移 | 中 | 中 | 以 `packages/bensz-nsfc/package.json`、模板元信息与 Git tag 三方校验，保持文档同步 |
| 过度抽象导致排错困难 | 高 | 中 | 对版式敏感项允许保留在项目覆盖层 |
| 文档与实现脱节 | 中 | 中 | 示例文件参与实际编译测试，避免只写不测 |
| 版本缓存目录持续膨胀 | 低 | 高 | `prune` 命令定期清理未引用版本；文档建议保留最近 2–3 个稳定版本 + 当前开发版本 |
