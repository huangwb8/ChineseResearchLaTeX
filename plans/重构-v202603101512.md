# NSFC LaTeX 模板重构计划

**版本**: v202603101512-optimized  
**创建时间**: 2026-03-10 15:12  
**更新时间**: 2026-03-14 （新增模板版本控制系统设计）  
**目标**: 将 `projects` 目录下的共性配置抽取为 `nsfc-common` LaTeX 包，采用**分层设计**保证灵活性，并提供 Python 脚本管理包的安装和更新。

---

## 一、项目背景

### 当前问题

1. **代码重复**：三个项目（General/Local/Young）的 `@config.tex` 大量内容相同。
2. **维护成本高**：修改配置时需要同步更新多个文件。
3. **一致性风险**：人工同步容易遗漏，导致模板行为不一致。
4. **用户诉求分层**：既有想“一行可用”的新手，也有想“深度接管”的高级用户。

### 重构目标

1. **DRY 原则**：消除重复配置，建立单一真相来源。
2. **易维护**：共性配置集中维护，项目差异可控覆盖。
3. **保持灵活性**：用户可在“开箱即用”和“深度定制”之间自由选择。
4. **用户友好**：提供类似 `pip install` 的安装体验。
5. **向后兼容**：不破坏现有用户的使用习惯，支持渐进式迁移。
6. **可验证**：重构后可通过编译与 PDF 对比验证版式一致性。

---

## 二、技术方案

### 2.1 设计哲学：分层架构

采用**三层配置架构**，在提升可维护性的同时保持用户的自定义灵活性：

```text
┌─────────────────────────────────────────────────────┐
│ 第 3 层：项目自定义层 (@config.tex)                 │
│ - 优先级：最高                                      │
│ - 用途：承接项目差异与少量必要覆盖                  │
└─────────────────────────────────────────────────────┘
                        ↓ 覆盖
┌─────────────────────────────────────────────────────┐
│ 第 2 层：项目类型层 (nsfc-common 包选项)            │
│ - 优先级：中等                                      │
│ - 用途：按 general/local/young 应用差异化配置       │
└─────────────────────────────────────────────────────┘
                        ↓ 继承
┌─────────────────────────────────────────────────────┐
│ 第 1 层：基础配置层 (nsfc-common 核心)              │
│ - 优先级：最低                                      │
│ - 用途：提供共性配置（宏包、字体、行距等）          │
└─────────────────────────────────────────────────────┘
```

借鉴已有成熟 texmf 包的经验，本次重构进一步强调以下几点：

- **用户入口保持极简**：用户仍然只接触 `main.tex -> @config.tex`
- **内部结构模块化**：按 layout / typography / headings / bibliography 等关注点拆包
- **参数与实现分离**：项目类型差异放入 profile 文件，模块只消费参数
- **分发方式标准化**：安装脚本之外，补充 TDS zip 打包能力

### 2.2 使用原则

本次重构改为**单一路径**，不再提供“完全包 / 最小包 / 混合 / 独立”四种模式：

- 用户继续沿用现有入口：`main.tex -> extraTex/@config.tex`
- `@config.tex` 从“完整配置大文件”收敛为“项目薄封装”
- 共性配置统一下沉到 `nsfc-common.sty`
- 项目差异仅保留在各自 `@config.tex` 中

这样可以避免模式分叉、降低测试矩阵，并保持现有用户的使用习惯不变。

### 2.4 包设计

#### 核心原则

- `nsfc-common.sty` 只作为**薄入口**，避免把所有逻辑堆进一个文件。
- `nsfc-core.sty` 负责**选项解析、profile 加载、模块调度**。
- 各功能模块只负责一个关注点，避免“大一统配置文件”继续膨胀。
- 项目类型差异通过 profile 文件集中定义，形成清晰的单一真相来源。
- `@config.tex` 仍然保留，但职责收敛为项目层薄封装。
- 不再额外引入 `@config-override.tex`、模式切换开关或多套入口。
- 优先保证默认路径简单稳定，而不是追求“所有层都可自由拼装”。

#### 选项系统

为了保留你原来设计的 `type=general` 用法，包内部建议显式采用 `kvoptions`（或等效 key-value 方案）实现，而不是只用基础 `\DeclareOption`。

建议支持的选项：

- `type=general|local|young`

这样既保留原设计中的项目类型分发能力，又避免把“灵活性”做成额外复杂度。

#### 内部结构约束

建议将公共包内部稳定拆成以下几层：

- **入口层**：`nsfc-common.sty`
- **核心层**：`nsfc-core.sty`
- **参数层**：`profiles/nsfc-profile-*.def`
- **功能层**：`nsfc-layout.sty`、`nsfc-typography.sty`、`nsfc-headings.sty`、`nsfc-bibliography.sty`

每层职责如下：

- 入口层只暴露包名，不承载具体版式细节
- 核心层只做“读选项、载入 profile、按顺序加载模块”
- 参数层只定义数字、开关和项目类型差异，不直接写实现逻辑
- 功能层只消费参数，不重新发明项目类型判断

#### 钩子与覆盖策略

保留用户覆盖钩子，但控制数量，优先保留以下几个：

- `\NSFCBeforePackages`
- `\NSFCAfterPackages`
- `\NSFCBeforeFonts`
- `\NSFCAfterFonts`
- `\NSFCUserOverride`

优化约束：

1. 钩子数量控制在最小必要集，避免“处处可插”的维护负担。
2. 对普通用户，不额外暴露“模式选择”；默认只维护 `@config.tex` 这一个项目入口。
3. 文档里只说明必要钩子，不把钩子包装成新的使用流派。

#### `nsfc-common.sty` 结构建议

```latex
\NeedsTeXFormat{LaTeX2e}
\ProvidesPackage{nsfc-common}[2026/03/10 v1.0.0 NSFC Common Package]

\RequirePackage{nsfc-core}
```

说明：

- `nsfc-common.sty` 应尽量薄，避免再次成为“新的大杂烩”。
- 真正的公共逻辑建议下沉到 `nsfc-core.sty` 与各子模块中。
- 版式敏感部分不要为了“抽得更干净”而硬统一，必要时仍留在项目层 `@config.tex` 中。

#### `nsfc-core.sty` 结构建议

```latex
\NeedsTeXFormat{LaTeX2e}
\ProvidesPackage{nsfc-core}[2026/03/10 v1.0.0 NSFC Core]

\RequirePackage{kvoptions}
\RequirePackage{etoolbox}
\SetupKeyvalOptions{family=NSFC,prefix=NSFC@}

\DeclareStringOption[general]{type}
\ProcessKeyvalOptions*

% 用户覆盖钩子
\providecommand{\NSFCBeforePackages}{}
\providecommand{\NSFCAfterPackages}{}
\providecommand{\NSFCBeforeFonts}{}
\providecommand{\NSFCAfterFonts}{}
\providecommand{\NSFCUserOverride}{}

% 载入项目 profile
\InputIfFileExists{profiles/nsfc-profile-\NSFC@type.def}{}{%
  \PackageError{nsfc-common}{Profile '\NSFC@type' not found}{}
}

% 按顺序载入模块
\RequirePackage{nsfc-layout}
\RequirePackage{nsfc-headings}
\RequirePackage{nsfc-typography}
\RequirePackage{nsfc-bibliography}
```

说明：

- 这里借鉴成熟包的做法，把“项目类型差异”与“实现逻辑”解耦。
- 如果后续需要新增 `重点项目` 等模板类型，只需新增 profile，而不是在一个大文件里继续堆 `ifthenelse`。

#### Profile 设计建议

三个 profile 文件应集中保存“项目级差异参数”，例如：

- 页边距
- 标题前后距
- 参考文献条目宽度
- 是否启用 `\raggedbottom`
- 其它确实存在项目差异的版式数值

原则：

- 共性数值放核心默认值
- 项目差异数值放 profile
- 项目私有、且强依赖正文结构的内容留在 `@config.tex`

这样可以把现在三个 `@config.tex` 之间的差异，逐步重组为“参数差异”，而不是“整段代码差异”。

### 2.5 使用示例

#### 推荐接入方式

```latex
\documentclass[12pt,UTF8,AutoFakeBold=2.5,a4paper,fontset=none]{ctexart}
\input{extraTex/@config.tex}

\begin{document}
% 内容
\end{document}
```

其中 `extraTex/@config.tex` 推荐收敛为：

```latex
\usepackage[type=general]{nsfc-common}
% 仅保留当前项目的少量差异化配置
```

不建议在首版引入新的用户命令（如 `\NSFCSetup{...}`）作为主要入口，避免增加学习成本。

### 2.7 Python 包管理器

#### 核心功能

1. **安装**：`python common/install.py install`
   - 检测 `TEXMFHOME`
   - 复制整个包目录到 `$TEXMFHOME/tex/latex/nsfc-common/`
   - 刷新文件数据库（如需要）
   - 保存版本信息

2. **升级**：`python common/install.py upgrade`
   - 检查当前版本与已安装版本
   - 备份旧版本
   - 安装新版本

3. **卸载**：`python common/install.py uninstall`
   - 删除安装的包文件
   - 清理版本信息

4. **状态查询**：`python common/install.py status`
   - 显示当前版本、已安装版本、安装路径

5. **打包分发**：`python common/scripts/build_tds_zip.py`
   - 生成可直接解压到本地 `texmf` 的 TDS 兼容压缩包
   - 作为 Release 附件分发

#### 技术细节

- **跨平台支持**：自动检测 Windows/macOS/Linux
- **版本管理**：在安装目录保存 `.version`
- **备份机制**：升级前备份到 `.backup`
- **错误处理**：提供清晰错误提示与回滚建议
- **兼容策略**：即便安装器不可用，仓库内模板仍保留现有入口结构，可通过仓库内分发方式直接编译
- **分发标准化**：优先按 TDS 目录结构组织，降低安装与发布复杂度
- **版本管理**：详见 2.8 节，安装器承担多版本管理的全部职责

### 2.8 模板版本控制系统

#### 设计动机

用户在撰写 NSFC 申请书时，往往需要长期停留在某一模板版本（避免格式意外变动影响已投稿版本的一致性），同时保留随时升级或回退到任意历史版本的能力。因此，版本控制是本次重构的核心功能之一，而非可选附加。

#### 设计原则

| 原则 | 说明 |
|------|------|
| **版本隔离** | 多个版本可同时缓存在本地，互不干扰 |
| **项目级锁定** | 每个项目可独立锁定所需版本，不受全局升级影响 |
| **来源统一** | GitHub Releases 作为唯一官方版本注册表 |
| **离线优先** | 本地缓存版本可脱网使用，无需每次联网 |
| **语义简洁** | 版本操作接近 pyenv/nvm 的使用体验 |

#### 版本注册表

以 GitHub Releases 作为版本注册表：

- 每个 Release 包含一个 TDS zip 附件（`nsfc-common-vX.Y.Z-tds.zip`）
- 安装器通过 GitHub API 获取可用版本列表并缓存到本地 `registry.json`
- 离线时使用已缓存的版本列表与本地已下载版本，无需联网

#### 本地存储结构

```text
~/.nsfc-common/                      # 版本管理器根目录
├── versions/                        # 已安装版本的本地缓存
│   ├── v1.0.0/                      # 某版本的完整包文件
│   │   ├── nsfc-common.sty
│   │   ├── nsfc-core.sty
│   │   └── ...
│   ├── v1.1.0/
│   └── v1.2.0/
├── registry.json                    # GitHub Releases 版本列表缓存
└── config.json                      # 记录当前全局激活版本
```

TEXMFHOME 中始终只保留**一个激活版本**的副本，由 `install.py use` 切换。

#### 项目级版本锁定

每个项目目录下可放置 `.nsfc-version` 锁文件，内容为版本号字符串：

```text
# projects/NSFC_General/.nsfc-version
v1.0.0
```

安装器读取此文件以：

1. 在 `install.py check` 时提示全局版本与项目期望版本是否一致
2. 在 `install.py pin` 时自动写入版本号

锁文件应加入版本控制（`git add`），确保团队协作时版本一致。

#### 版本命令（完整列表）

在现有 `install / upgrade / uninstall / status` 基础上新增：

| 命令 | 说明 |
|------|------|
| `install.py install --version v1.0.0` | 下载并缓存指定版本（不切换激活版本） |
| `install.py use v1.0.0` | 切换全局激活版本到指定版本 |
| `install.py list` | 列出所有本地已缓存的版本 |
| `install.py list --remote` | 从 GitHub 拉取并展示所有可用版本 |
| `install.py pin v1.0.0` | 在当前项目目录写入 `.nsfc-version` 锁文件 |
| `install.py unpin` | 移除当前项目的版本锁文件 |
| `install.py rollback` | 回退到上一个激活版本 |
| `install.py prune` | 清理未被任何项目锁定的旧版本本地缓存 |
| `install.py info v1.0.0` | 查看指定版本的详细元信息（发布日期、变更摘要等） |
| `install.py check` | 检查全局激活版本是否与当前项目锁定版本一致 |

#### 典型用户工作流

**场景 1：开始写作，锁定当前版本**

```bash
# 进入项目，安装当前最新版并锁定
python common/install.py install
python common/install.py pin v1.0.0     # 写入 .nsfc-version，提交到 Git
```

**场景 2：全局升级，不影响旧项目**

```bash
python common/install.py install --version v1.2.0
python common/install.py use v1.2.0    # 全局切换到新版
# NSFC_General/ 仍有 .nsfc-version = v1.0.0
python common/install.py check          # 提示版本不一致，需手动切换
```

**场景 3：临时切回旧版本编译旧项目**

```bash
cd projects/NSFC_General/
python common/install.py use $(cat .nsfc-version)  # 临时切到 v1.0.0
# ... 编译 ...
python common/install.py use v1.2.0    # 还原
```

**场景 4：清理本地缓存**

```bash
python common/install.py prune         # 只保留被 .nsfc-version 锁定的版本
```

#### LaTeX 包内版本声明

每个版本的 `nsfc-common.sty` 在包声明中嵌入版本号：

```latex
\ProvidesPackage{nsfc-common}[2026/03/10 v1.0.0 NSFC Common Package]
```

安装器通过解析此字符串验证已激活版本，无需额外 `.version` 文件。

## 四、风险评估与应对

### 4.1 技术风险

| 风险 | 影响 | 概率 | 应对措施 |
|------|------|------|----------|
| `type=general` 示例与实现不一致 | 高 | 中 | 显式引入 key-value 选项系统，统一示例与实现 |
| 钩子机制过多导致维护复杂 | 中 | 中 | 保留最小必要钩子集，不再额外设计 override 分支 |
| 公共包抽取过度导致版式回归 | 高 | 中 | 先抽稳定共性，版式敏感项继续留在项目层 |
| 模块加载顺序不当引发副作用 | 高 | 中 | 在核心层固定模块加载顺序，并用编译测试覆盖 |
| profile 参数与模块实现漂移 | 中 | 中 | 以 profile 文件为参数单一事实来源，文档同步引用 |
| 仓库内直编与安装器分发路径不一致 | 高 | 中 | 在首轮实现中先固定仓库内分发策略，再补安装器适配 |
| 跨平台字体差异影响 PDF 对比 | 中 | 中 | 将像素级比对限定在同环境，跨平台采用结构性校验 |
| 版本切换后 TEXMFHOME 缓存未刷新导致旧版本仍生效 | 高 | 中 | 切换时强制执行 `mktexlsr`，并在编译前提示用户验证激活版本 |
| GitHub API 限流或不可达导致版本拉取失败 | 中 | 中 | 离线降级策略：优先使用本地 `registry.json` 缓存，并给出友好提示 |
| `.nsfc-version` 锁文件与实际 TEXMFHOME 版本不一致 | 高 | 中 | `install.py check` 在编译前自动检测并警告，避免静默编译错误版本 |
| 多项目并发使用不同版本（全局只有一个激活版本）| 中 | 高 | 文档明确说明全局单激活限制；进阶方案提供项目本地 texmf 目录覆盖路径 |

### 4.2 用户体验风险

| 风险 | 影响 | 概率 | 应对措施 |
|------|------|------|----------|
| 用户不愿安装 Python 脚本 | 中 | 中 | 保留手动安装说明与仓库内直编路径 |
| 用户担心失去灵活性 | 中 | 中 | 保留项目层 `@config.tex`，只抽取稳定共性 |
| 用户默认路径被打断 | 高 | 低 | 保持 `main.tex -> @config.tex` 入口不变 |
| 版本控制概念对新手过于复杂 | 中 | 中 | 默认路径无需理解版本控制；锁文件和 use 命令作为可选进阶功能文档化 |

### 4.3 维护风险

| 风险 | 影响 | 概率 | 应对措施 |
|------|------|------|----------|
| 安装器与包版本漂移 | 中 | 中 | 以 `common/package.json` 为安装器版本源，保持文档同步 |
| 过度抽象导致排错困难 | 高 | 中 | 对版式敏感项允许保留在项目覆盖层 |
| 文档与实现脱节 | 中 | 中 | 示例文件参与实际编译测试，避免只写不测 |
| 版本缓存目录持续膨胀 | 低 | 高 | `prune` 命令定期清理未引用版本；文档建议保留最近 2–3 个版本 |
