# NSFC LaTeX 模板重构计划

**版本**: v202603101512-optimized  
**创建时间**: 2026-03-10 15:12  
**更新时间**: 2026-03-10 16:28  
**目标**: 将 `projects` 目录下的共性配置抽取为 `nsfc-common` LaTeX 包，采用**分层设计**保证灵活性，并提供 Python 脚本管理包的安装和更新。

---

## 一、项目背景

### 当前问题

1. **代码重复**：三个项目（General/Local/Young）的 `@config.tex` 大量内容相同。
2. **维护成本高**：修改配置时需要同步更新多个文件。
3. **一致性风险**：人工同步容易遗漏，导致模板行为不一致。
4. **用户诉求分层**：既有想“一行可用”的新手，也有想“深度接管”的高级用户。

### 重构目标

1. **DRY 原则**：消除重复配置，建立单一真相来源。
2. **易维护**：共性配置集中维护，项目差异可控覆盖。
3. **保持灵活性**：用户可在“开箱即用”和“深度定制”之间自由选择。
4. **用户友好**：提供类似 `pip install` 的安装体验。
5. **向后兼容**：不破坏现有用户的使用习惯，支持渐进式迁移。
6. **可验证**：重构后可通过编译与 PDF 对比验证版式一致性。

## 二、技术方案

### 2.1 设计哲学：分层架构

采用**三层配置架构**，在提升可维护性的同时保持用户的自定义灵活性：

```text
┌─────────────────────────────────────────────────────┐
│ 第 3 层：用户自定义层 (@config.tex / override 文件)   │
│ - 优先级：最高                                      │
│ - 用途：覆盖包的默认配置，实现深度定制              │
└─────────────────────────────────────────────────────┘
                        ↓ 覆盖
┌─────────────────────────────────────────────────────┐
│ 第 2 层：项目类型层 (nsfc-common 包选项)            │
│ - 优先级：中等                                      │
│ - 用途：按 general/local/young 应用差异化配置       │
└─────────────────────────────────────────────────────┘
                        ↓ 继承
┌─────────────────────────────────────────────────────┐
│ 第 1 层：基础配置层 (nsfc-common 核心)              │
│ - 优先级：最低                                      │
│ - 用途：提供共性配置（宏包、字体、行距等）          │
└─────────────────────────────────────────────────────┘
```

### 2.2 使用模式

用户可以根据需求选择不同的使用模式：

| 模式 | 适用人群 | 灵活性 | 维护成本 | 使用方式 |
|------|---------|--------|---------|---------|
| **完全包模式** | LaTeX 新手、追求稳定 | ★☆☆ | ★★★ | `\usepackage[type=general]{nsfc-common}` |
| **最小包模式** | 高级用户、深度定制 | ★★★ | ★☆☆ | `\usepackage[type=general,minimal]{nsfc-common}` + 自定义 |
| **混合模式** | 大多数用户（推荐） | ★★☆ | ★★☆ | 包 + `@config-override.tex` 覆盖 |
| **独立模式** | 完全自主控制 | ★★★ | ★☆☆ | 不使用包，保留完整 `@config.tex` |

### 2.4 包设计

#### 核心原则

- `nsfc-common.sty` 负责**共性能力 + 项目类型差异化分发**。
- `@config.tex` 仍然保留，作为用户层入口与独立模式兜底。
- `@config-override.tex` 作为混合模式推荐覆盖入口。
- 包内部尽量提供稳定扩展点，但避免为了“极致灵活”破坏可读性。

#### 选项系统

为了保留你原来设计的 `type=general` 用法，包内部建议显式采用 `kvoptions`（或等效 key-value 方案）实现，而不是只用基础 `\DeclareOption`。

建议支持的选项：

- `type=general|local|young`
- `minimal=true|false`
- `nopackages=true|false`
- `nofonts=true|false`
- `notitles=true|false`

这样既保留原设计，又避免示例代码与真实实现脱节。

#### 钩子与覆盖策略

保留用户覆盖钩子，但控制数量，优先保留以下几个：

- `\NSFCBeforePackages`
- `\NSFCAfterPackages`
- `\NSFCBeforeFonts`
- `\NSFCAfterFonts`
- `\NSFCUserOverride`

优化约束：

1. 钩子数量控制在最小必要集，避免“处处可插”的维护负担。
2. 文档中明确两种写法：
   - **包前定义**：使用 `\newcommand`；
   - **包后覆盖**：使用 `\renewcommand` 或 `\input{extraTex/@config-override.tex}`。
3. 对大多数用户，优先推荐 `@config-override.tex`，而非直接改钩子。

#### `nsfc-common.sty` 结构建议

```latex
\NeedsTeXFormat{LaTeX2e}
\ProvidesPackage{nsfc-common}[2026/03/10 v1.0.0 NSFC Common Package]

\RequirePackage{kvoptions}
\SetupKeyvalOptions{family=NSFC,prefix=NSFC@}

% 项目类型
\DeclareStringOption[general]{type}

% 灵活性选项
\DeclareBoolOption[false]{minimal}
\DeclareBoolOption[false]{nopackages}
\DeclareBoolOption[false]{nofonts}
\DeclareBoolOption[false]{notitles}

\ProcessKeyvalOptions*

% 用户覆盖钩子
\providecommand{\NSFCBeforePackages}{}
\providecommand{\NSFCAfterPackages}{}
\providecommand{\NSFCBeforeFonts}{}
\providecommand{\NSFCAfterFonts}{}
\providecommand{\NSFCUserOverride}{}

% 基础配置
\NSFCBeforePackages
% ... 加载共性宏包
\NSFCAfterPackages

% 字体配置
\NSFCBeforeFonts
% ... 字体逻辑
\NSFCAfterFonts

% 项目差异化配置
\ifthenelse{\equal{\NSFC@type}{general}}{...}{}
\ifthenelse{\equal{\NSFC@type}{local}}{...}{}
\ifthenelse{\equal{\NSFC@type}{young}}{...}{}

% 用户最终覆盖
\NSFCUserOverride
```

说明：

- 这里不强求逐行照抄实现，重点是**让示例语法与实现机制一致**。
- 版式敏感部分不要为了“抽得更干净”而硬统一，必要时仍留在项目层覆盖。

### 2.5 使用示例

#### 模式 1：完全包模式

```latex
\documentclass{article}
\usepackage[type=general]{nsfc-common}

\begin{document}
% 内容
\end{document}
```

#### 模式 2：最小包模式

```latex
\documentclass{article}
\usepackage[type=general,minimal=true,nopackages=true,nofonts=true,notitles=true]{nsfc-common}

\usepackage{xcolor}
\usepackage{graphicx}
\setCJKmainfont{Noto Serif CJK SC}
\titleformat{\section}{...}{}{0pt}{}

\begin{document}
% 内容
\end{document}
```

#### 模式 3：混合模式

```latex
\documentclass{article}

\newcommand{\NSFCAfterPackages}{%
  \usepackage{tikz}
  \usepackage{pgfplots}
}

\usepackage[type=general]{nsfc-common}
\input{extraTex/@config-override.tex}

\begin{document}
% 内容
\end{document}
```

#### 模式 4：独立模式

```latex
\documentclass{article}
\input{extraTex/@config.tex}

\begin{document}
% 内容
\end{document}
```

### 2.7 Python 包管理器

#### 核心功能

1. **安装**：`python common/install.py install`
   - 检测 `TEXMFHOME`
   - 复制 `.sty` 文件到 `$TEXMFHOME/tex/latex/nsfc-common/`
   - 刷新文件数据库（如需要）
   - 保存版本信息

2. **升级**：`python common/install.py upgrade`
   - 检查当前版本与已安装版本
   - 备份旧版本
   - 安装新版本

3. **卸载**：`python common/install.py uninstall`
   - 删除安装的包文件
   - 清理版本信息

4. **状态查询**：`python common/install.py status`
   - 显示当前版本、已安装版本、安装路径

#### 技术细节

- **跨平台支持**：自动检测 Windows/macOS/Linux
- **版本管理**：在安装目录保存 `.version`
- **备份机制**：升级前备份到 `.backup`
- **错误处理**：提供清晰错误提示与回滚建议
- **兼容策略**：即便安装器不可用，仓库内模板仍可直接编译

## 四、风险评估与应对

### 4.1 技术风险

| 风险 | 影响 | 概率 | 应对措施 |
|------|------|------|----------|
| `type=general` 示例与实现不一致 | 高 | 中 | 显式引入 key-value 选项系统，统一示例与实现 |
| 钩子机制过多导致维护复杂 | 中 | 中 | 保留最小必要钩子集，优先推荐 override 文件 |
| 多模式测试矩阵膨胀 | 高 | 中 | 明确模式优先级：混合模式为默认主路径，其他模式做回归验证 |
| 包更新导致兼容性问题 | 高 | 中 | 保留独立模式兜底，安装器支持回滚与版本检查 |
| 跨平台字体差异影响 PDF 对比 | 中 | 中 | 将像素级比对限定在同环境，跨平台采用结构性校验 |

### 4.2 用户体验风险

| 风险 | 影响 | 概率 | 应对措施 |
|------|------|------|----------|
| 用户不愿安装 Python 脚本 | 中 | 中 | 保留手动安装说明与独立模式 |
| 用户不理解四种模式差异 | 中 | 高 | 在 README 与迁移指南中给出决策树与示例 |
| 用户担心失去灵活性 | 高 | 中 | 通过钩子 + override + 独立模式兜底保证自由度 |
| 用户默认路径被打断 | 高 | 低 | 保持旧入口可用，默认推荐混合模式 |

### 4.3 维护风险

| 风险 | 影响 | 概率 | 应对措施 |
|------|------|------|----------|
| 安装器与包版本漂移 | 中 | 中 | 以 `common/package.json` 为安装器版本源，保持文档同步 |
| 过度抽象导致排错困难 | 高 | 中 | 对版式敏感项允许保留在项目覆盖层 |
| 文档与实现脱节 | 中 | 中 | 示例文件参与实际编译测试，避免只写不测 |
