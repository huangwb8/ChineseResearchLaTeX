# NSFC LaTeX 模板重构计划

**版本**: v202603101512-revised  
**创建时间**: 2026-03-10 15:12  
**更新时间**: 2026-03-10 16:10  
**当前结论**: 先做**仓库内去重**，保持现有模板入口与用户工作流不变；是否升级为可全局安装的 `.sty` 包，放到后续阶段再决策。

---

## 背景

当前三套 NSFC 模板的入口保持一致：`main.tex` 直接加载各自的 `extraTex/@config.tex`，例如 `projects/NSFC_General/main.tex:18`。

现状有两个同时成立的事实：

- 三套 `@config.tex` 存在大量共性，重复维护成本高。
- 三套模板又不是“完全一样”，其中页边距、标题 spacing、参考文献细节、圈号样式等都属于**版式敏感差异**，不能粗暴合并。

这意味着本次重构的核心不是“把配置硬塞成一个通用包”，而是：

- 先找出**稳定共性**；
- 再保留**项目差异**；
- 最后在**不改变用户入口**的前提下完成去重。

---

## 重构原则

本计划遵循项目既有原则，重点强调以下几条：

- **正确性优先**：版式一致性优先于架构美观。
- **KISS**：优先最小可行重构，不把 v1 做成一个“包管理系统”。
- **YAGNI**：没有明确用户价值前，不引入全局安装器、四种模式、GUI、在线更新等扩展能力。
- **向后兼容**：现有用户继续使用 `main.tex + extraTex/@config.tex` 的方式，不要求学习新入口。
- **可回滚**：任何阶段都要能快速回退到重构前状态。

## 本次范围

### 本次要做

- 梳理三套 `@config.tex` 的共性项与差异项。
- 在仓库内提取一个共享基础配置文件。
- 将三套 `@config.tex` 改成“共享基座 + 少量项目覆盖”的薄包装结构。
- 保持三套 `main.tex` 的默认用法不变，尽量不改或只做最小修改。
- 用现有三套模板完成编译验证与版式比对。
- 更新必要文档与根级 `CHANGELOG.md`。

### 本次不做

- 不做 TEXMF 全局安装。
- 不做 Python 安装/升级/卸载脚本。
- 不做 `package.json` 一类与 TeX 生态不匹配的元信息文件。
- 不设计“四种使用模式”。
- 不引入复杂钩子系统。
- 不承诺 CTAN 发布。

这些能力只有在 **仓库内去重方案稳定** 后，才进入下一阶段评估。

### 设计约束

- `projects/_shared/nsfc-base.tex` 只承载**稳定共性**。
- 各项目 `extraTex/@config.tex` 只保留：
  - 项目类型标识；
  - 页边距等项目差异；
  - 少量必须保留在本项目内的版式细节。
- `main.tex` 继续使用 `\input{extraTex/@config.tex}`，避免改变用户心智。
- 不额外保留一份“完整旧版 `@config.tex` 副本”；回滚依赖 Git 即可，避免制造新的重复源。

## 建议实现方式

### 共享层

在 `projects/_shared/nsfc-base.tex` 中集中维护以下内容：

- 通用宏包加载；
- 颜色定义；
- 对象名重定义；
- 字号命令；
- 字体基础配置；
- 行距与段落基础设置；
- 图片、表格、超链接基础配置；
- 参考文献环境的共性部分；
- 通用辅助命令（如正文段落控制命令）。

### 项目层

每个 `extraTex/@config.tex` 负责两件事：

1. 引入共享基座；
2. 写入本项目独有的覆盖项。

建议形态如下：

```latex
% projects/NSFC_General/extraTex/@config.tex
\def\NSFCProjectType{general}
\input{../../_shared/nsfc-base.tex}

% General 独有配置
\geometry{left=3.00cm,right=2.75cm,top=2.63cm,bottom=2.96cm}
\newcommand{\ssssubtitle}[1]{...}
```

说明：

- 若某些配置需要“先定义项目类型、后在共享层分支判断”，可以在共享层中读取 `\NSFCProjectType`。
- 若某些配置属于**后置覆盖更安全**的类型，则直接在项目 `@config.tex` 尾部覆盖，不必强行参数化。
- 不在 v1 使用 `\DeclareOption`、`type=general`、自定义钩子等包级机制，避免把简单问题升级成 TeX 包设计问题。

## 识别与拆分规则

在正式改动前，先把配置分成三类：

### 可直接抽取的稳定共性

- 三套模板完全一致，或语义一致且可统一。
- 对版式没有项目依赖。
- 抽取后不会改变任何一套模板的现有行为。

### 应保留在项目层的差异项

- 页边距；
- 特定圈号命令实现；
- 标题间距的细微差异；
- Young/General/Local 的个别参考文献与页面策略差异；
- 任何一项“统一后可能影响像素级对齐”的设置。

### 暂不抽取的高风险项

- 一旦抽取就需要引入大量条件分支；
- 抽取后可读性明显下降；
- 无法在短时间内通过 PDF 对比确认安全性。

这类项宁可先保留重复，也不要为 DRY 牺牲稳定性。

## 实施阶段

### 阶段 A：差异盘点

产出一份明确清单：

- 哪些内容 100% 可抽；
- 哪些内容必须按项目保留；
- 哪些内容存在争议，暂缓处理。

完成标准：

- 共性/差异边界清楚；
- 不凭“感觉 90% 相同”推进；
- 后续每一项抽取都有依据。

### 阶段 B：提取共享基座

- 新建 `projects/_shared/nsfc-base.tex`。
- 先迁移低风险共性。
- 保证共享层本身不依赖特定项目路径和临时开关。

完成标准：

- 三套模板都能成功加载共享层；
- 共享层职责单一；
- 不出现为了“通用”而增加的大量条件判断。

### 阶段 C：瘦身三套 `@config.tex`

- General、Local、Young 三套 `@config.tex` 改成薄包装。
- 每套文件只保留真正的差异与必要说明。
- 尽量不改 `main.tex`；如必须改动，也只允许最小调整。

完成标准：

- 用户仍然只需维护原来的模板目录；
- 模板入口不变；
- 新结构一眼能看懂“共性在哪、差异在哪”。

### 阶段 D：编译验证

按项目规定执行编译链：

```text
xelatex → bibtex → xelatex → xelatex
```

验证对象：

- `projects/NSFC_General/`
- `projects/NSFC_Local/`
- `projects/NSFC_Young/`

验证内容：

- 是否能完整编译通过；
- 页数是否一致；
- 章节标题分页是否一致；
- 参考文献块位置与换行是否异常；
- 关键页面视觉是否与重构前保持一致。

说明：

- “像素级对比”仅在**同一系统、同一字体集、同一 TeX 环境**下作为强校验。
- 跨平台只要求版式语义一致，不强求渲染像素完全相同。

### 阶段 E：文档与记录

- 更新根级 `CHANGELOG.md`。
- 如有必要，在 `README.md` 中补充模板结构说明。
- 本次不计划修改 `AGENTS.md` 与 `CLAUDE.md`；只有当工作流规则真正变化时才修改，且两者需保持一致。

## 验收标准

### 结构验收

- 重复配置显著减少；
- 共享层与项目层职责清晰；
- 没有制造第二套新的重复源。

### 兼容性验收

- 三套模板保持原有入口；
- 旧用户不需要学习新命令；
- 不要求额外安装任何本地包。

### 版式验收

- 三套模板均可按四步法稳定编译；
- 关键页面与重构前一致；
- 未引入新的 LaTeX 警告或引用异常。

### 维护性验收

- 之后修改共性配置时，只需优先修改共享层；
- 项目差异仍可在各自 `@config.tex` 中局部维护；
- 新贡献者能快速理解结构。

## 风险与应对

| 风险 | 影响 | 判断 | 应对 |
|------|------|------|------|
| 误把版式敏感项抽到共享层 | 高 | 高 | 先做差异盘点，再抽取；高风险项宁可暂不抽 |
| 为了“优雅”引入过多条件分支 | 中 | 高 | 共享层只收稳定共性，避免过度参数化 |
| 修改 `main.tex` 破坏用户工作流 | 高 | 中 | v1 原则上不改 `main.tex` |
| 跨平台渲染差异导致误判 | 中 | 中 | 将像素比对限定在同环境；跨平台用结构性检查 |
| 参考文献环境合并后细节漂移 | 高 | 中 | 单独抽样验证含引用页面与 bibliography 页面 |

---

## 回滚策略

- 每完成一个阶段就单独提交或至少单独保存差异。
- 如共享层引入后出现难以快速定位的问题，优先整体回退该阶段，不在失稳状态下继续叠加修改。
- 不通过额外保留“旧版副本文件”来回滚，统一依赖 Git 历史。

---

## 后续阶段（本次不实现）

只有在以下条件满足时，才评估进入下一阶段：

- 仓库内去重方案已稳定；
- 至少完成一轮完整编译验证；
- 用户确实存在“仓库外复用共享配置”的需求。

届时再单独立项讨论：

- 是否升级为 `.sty` 包；
- 是否提供本地安装脚本；
- 是否需要更正式的元信息与发布机制。

这部分必须作为**独立计划**处理，不与本次重构混做。

**先在仓库内部完成稳定去重，保持用户入口不变，把复杂度留给真正出现的需求。**
